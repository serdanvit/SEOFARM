# SEO FARM — аудит соответствия архитектуре v5

Дата проверки: 2026-03-14.

## TL;DR

**Вердикт:** система в репозитории **в целом соответствует** целевой архитектуре v5 по крупным блокам (3 агента, планировщик, OAuth, AI+fallback, БД, ключевые API), но есть несколько критичных расхождений, из-за которых часть сценариев может ломаться на «чистой» БД.

## Что совпадает с архитектурой

- Есть 3 реализованных агента:
  - `agents/vk_groups/*` (создание, прогрев, репосты, активность).
  - `agents/comment_monitor/monitor.py` (мониторинг лидов + Telegram).
  - `agents/article_publisher/publisher.py` (публикация статей).
- Есть универсальный DB-адаптер SQLite/PostgreSQL через `DB_URL`.
- Есть шифрование токенов (PBKDF2 + HMAC + потоковый XOR).
- Есть scheduler с:
  - `_active_tasks` защитой от двойного запуска,
  - `_reset_stuck_tasks()`,
  - ограничением выборки `LIMIT 3` и периодом 30 секунд.
- Есть AI-слой через Ollama и fallback на шаблоны.
- Есть OAuth для VK (`client_id = 2685278`) и базовая интеграция Яндекса.
- Есть campaign flow: парсинг сайта → анализ ниши → генерация ключей → планирование задач.

## Ключевые расхождения и замечания (передать агенту)

### 1) Критично: несоответствие схемы БД и кода по `vk_groups`

В коде активно используются поля:
- `vk_groups.posts_count`
- `vk_groups.discussions_count`

Но в `core/db.py:init_schema()` такие столбцы **не создаются**.

**Риск:**
- падения на SQL-запросах в `/api/positions`, `warmup.py`, `creator.py`;
- нестабильная работа на новой базе (без «исторических» миграций).

**Что поручить агенту:**
- добавить миграцию/авто-добавление недостающих полей;
- выровнять модель `vk_groups` с реальным использованием в коде;
- добавить smoke-тест на `/api/positions` на пустой новой БД.

### 2) API-роуты: формально не 56 уникальных

В `app.py` декораторов `@app.route` — 57, но уникальных путей — 55 (есть алиасы, например `groups/start` и `create-now` ведут на один обработчик).

**Риск:** путаница в документации и ожиданиях команды.

**Что поручить агенту:**
- синхронизировать спецификацию API с фактическим списком эндпоинтов;
- указать, где алиас, а где отдельный маршрут.

### 3) Wordstat: в архитектуре заявлена интеграция, но в pipeline используется ограниченно

Wordstat-модуль реализован, однако основной поток `run_campaign` полагается прежде всего на AI/fallback генерацию ключей; прямое обязательное обогащение частотностью Wordstat в pipeline не является жёсткой частью шага.

**Риск:** расхождение ожиданий по качеству/валидности НЧ-ключей.

**Что поручить агенту:**
- сделать Wordstat-обогащение опциональным, но явно управляемым флагом настройки (`use_wordstat=true/false`),
- логировать «какая доля ключей прошла через реальную частотность».

### 4) PostgreSQL режим: технически поддержан, но dependency не в активном requirements

`psycopg2-binary` закомментирован в `requirements.txt`.

**Риск:** «переключение одной строкой» не сработает без доп. шага установки.

**Что поручить агенту:**
- добавить установку PG-драйвера в отдельный extras/requirements-профиль;
- в startup-check вывести явную ошибку-консультацию, если выбран PostgreSQL, но драйвер не установлен.

### 5) Небольшая структурная разница с документом

В репозитории есть `agents/vk_groups/content_gen.py` (доп. модуль), которого нет в текстовой структуре v5.

**Риск:** минимальный, но документ устаревает.

**Что поручить агенту:**
- обновить раздел «Структура файлов», чтобы он был 1:1 к факту.

## Идеи по усилению архитектуры (мои предложения)

1. **Слой миграций БД (обязательно)**
   - Даже с SQLite нужен минимальный migration runner (`schema_version` + up/down скрипты).
   - Это уберёт класс ошибок «в коде колонка есть, в БД нет».

2. **Единый реестр API (single source of truth)**
   - Генерировать список маршрутов автоматически (например, health endpoint `/api/meta/routes`).
   - На основе него строить документацию фронта и QA-чеклист.

3. **Контракты задач в scheduler**
   - Ввести JSON-schema для `tasks.payload` по каждой паре `agent/type`.
   - Валидировать payload перед запуском — меньше «тихих» падений.

4. **Надёжность внешних API**
   - Для VK/Telegram/Telegra.ph добавить унифицированный retry/backoff helper и метрики ошибок по кодам.

5. **Наблюдаемость**
   - Добавить `/api/health` с проверками:
     - БД,
     - Ollama,
     - наличие активных токенов,
     - состояние scheduler.

6. **Тест-минимум для релизов**
   - smoke-набор:
     - инициализация пустой БД,
     - запуск app,
     - проверка ключевых endpoint’ов,
     - тест на создание/выполнение одной задачи каждого агента.

## Приоритеты для ближайшего спринта

2. **P0:** синхронизировать API-спеку с фактом.
3. **P1:** включить управляемый Wordstat-режим + прозрачные логи происхождения ключей.
4. **P1:** оформить PostgreSQL dependency-профиль.
5. **P2:** health endpoint + smoke-тесты.
